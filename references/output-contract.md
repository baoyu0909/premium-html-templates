# Output Contract

## 目标

输出必须不是“只像一点点”，而是要在下面三层都接近参考案例：

1. 视觉层
2. 布局层
3. 信息组织层

## 最低交付标准

- 先明确输出模式：`deck` 或 `single-page`
- 有明确的风格原型归属
- 有完整页面结构，不是散装模块拼贴
- 有统一色彩 token
- 有统一标题等级
- 有统一卡片规则
- 有统一边距和网格节奏

## HTML 交付规则

- 必须能直接打开运行
- 样式尽量内聚在单文件内，方便复用和转交
- 若使用外部资源，只使用稳定 CDN
- 若内容是 `deck`，必须保证 16:9 视觉稳定，不能靠内部滚动或缓冲区兜底
- 若内容是 `single-page`，页面和卡片可随内容自然增长，不要强套 16:9
- 默认不要生成额外的“展示型外壳”

这里的“展示型外壳”包括：

- 顶部测试说明条
- 悬浮总导航
- 为了解释风格来源而存在的引导文案区
- 与实际页面内容无关的总览封套

除非用户明确要求做总览、目录导航或多风格对比页，否则页面应从正式内容直接开始。

## 高贴合生成规则

生成时至少同步下面五项中的四项：

- 背景氛围
- 标题样式
- 卡片边界形式
- 章节切换结构
- 色彩主次关系

如果只同步配色，没有同步结构，不算高复刻。

## 内容改写规则

- 先重组内容，再写 UI
- 先选 mode，再选 variant
- 每个模块都要有角色：
  - 结论
  - 证据
  - 对比
  - 流程
  - 表格
  - 占位视觉
- 不要把所有内容都写成同一种卡片
- 不要为了“解释这是测试页”而添加额外说明模块
- `deck` 模式下，如果内容超量，必须拆页或删减，不得缩小到不可读或裁切
- `single-page` 模式下，如果内容超量，应拆 section 或拆卡片，不得强制塞进一屏

## Mode 约束

- `deck` 模式读取并遵守 [mode-deck.md](mode-deck.md)
- `single-page` 模式读取并遵守 [mode-single-page.md](mode-single-page.md)
- variant 文件只负责视觉语言，不能覆盖 mode 的尺寸和溢出规则

## 扩展规则

新增变体时，优先复制当前 reference 模式，而不是修改旧变体定义。

推荐扩展格式：

- `variant-xxx.md`
- 写清楚内部风格来源或风格语义
- 写清楚适用任务
- 写清楚 token 和布局骨架
- 写清楚禁止事项
